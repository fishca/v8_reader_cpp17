# Алгоритм открытия файла CF (v8_reader)

Данный документ описывает реальный алгоритм чтения файлов конфигурации 1С (.cf, .cfu), основанный на анализе исходного кода репозитория `v8_reader` (C++ Builder).

## 1. Точка входа (UI)

Процесс начинается в главном окне приложения.

- **Файл**: `src/MainUnit.cpp`
- **Класс**: `TMainForm`
- **Метод**: `void __fastcall TMainForm::ActionFileOpenExecute(TObject *Sender)`

**Логика**:
1. Вызывается стандартный диалог открытия файла (`OpenDialog`).
2. При выборе файла вызывается метод `OpenFile(OpenDialog->FileName)`.

## 2. Инициализация объекта каталога

В методе `TMainForm::OpenFile` происходит создание основного объекта для работы с файлом.

- **Класс**: Обертка над каталогом (в коде `v8_reader` используется класс, работающий с внутренней структурой каталога, часто обозначаемый как `v8catalog` или аналогичный в зависимости от версии).
- **Действия**:
    1. Очищается текущее дерево отображения.
    2. Создается новый объект для чтения файла.
    3. Определяется тип файла:
       - Если расширение `.cfu` (файл обновления), применяется предварительная распаковка потока deflate.
       - Иначе файл обрабатывается как обычный контейнер.
    4. Открывается поток `TFileStream`.
    5. Вызывается метод инициализации (например, `initialize`), который определяет версию формата.

## 3. Определение версии формата и смещений

Алгоритм поддерживает два основных формата хранения:

- **Format 15** (до платформы 8.3.15):
    - Используются 32-битные адреса (`uint32_t`).
    - Базовое смещение заголовка: `0`.
- **Format 16** (с платформы 8.3.16):
    - Используются 64-битные адреса (`uint64_t`).
    - Базовое смещение заголовка: `0x1359` (специфический оффсет для новых версий).

**Структуры заголовков**:
- `catalog_header` (для Format 15)
- `catalog_header8316` (для Format 16)

## 4. Чтение таблицы размещения файлов (FAT)

После чтения заголовка каталога алгоритм переходит к чтению таблицы размещения (аналог FAT).

**Процесс**:
1. Из заголовка каталога извлекается количество файлов и адрес таблицы.
2. Читается массив записей FAT. Каждая запись содержит:
    - Смещение к заголовку файла (`header_offset`).
    - Смещение к данным файла (`data_offset`).
    - Размер данных.
3. Для каждой записи FAT создается внутренний объект файла (`v8file`).

## 5. Чтение заголовков файлов и имен

Для каждой записи в FAT выполняется чтение метаданных файла:

1. Поток позиционируется на `header_offset`.
2. Читается заголовок файла, содержащий:
    - Имя файла (в кодировке UTF-16LE).
    - Время создания и модификации.
3. Имя преобразуется в строку Unicode для отображения в интерфейсе.

## 6. Чтение имени конфигурации (Спецобработка)

Для получения человекопонятного имени конфигурации выполняется специальная последовательность:

1. Открывается служебный файл с именем `version` (ищется в каталоге по имени).
2. Его содержимое читается и передается в функцию `get_treeFromV8file`.
3. Из полученного дерева извлекается GUID корневого элемента метаданных.
4. По этому GUID находится соответствующий файл в каталоге.
5. Из этого файла извлекается имя конфигурации.

## 7. Парсинг текстового представления 1С

Основное содержание файлов метаданных хранится в специфическом текстовом формате 1С.

**Процесс**:
1. Данные файла читаются из потока (при необходимости собираются из блоков).
2. Бинарные данные конвертируются в текст (Unicode).
3. Текст передается в парсер `parse_1Ctext`.
4. Парсер строит объектное дерево (`tree`), представляющее структуру метаданных (справочники, документы, реквизиты и т.д.).

**Важно**: В отличие от распространенного заблуждения, в данной реализации **не используется** прямая поблочная распаковка zlib для каждого элемента внутри основного цикла чтения каталога. Сжатие может применяться на уровне всего файла `.cfu` или специфических блоков, но основная логика — это навигация по FAT и чтение сырых данных файлов.

## 8. Построение дерева интерфейса

После первичного сканирования каталога:

1. Создается список всех найденных файлов.
2. Заполняется компонент `VirtualStringTree` (или аналогичный).
3. Реализована **ленивая загрузка**: данные конкретного файла читаются и парсятся только при раскрытии узла или выборе элемента пользователем.

## 9. Ключевые отличия от типовых описаний

При анализе кода `v8_reader` выявлены следующие особенности, которые часто упускаются:

- **Отсутствие структур `stBlockHeader` с цепочками страниц** в основном цикле чтения каталога. Файлы в каталоге адресуются напрямую через FAT.
- **Использование текстового парсера**. Метаданные хранятся не в бинарном виде, а в текстовом представлении 1С, которое требует специального парсинга (`parse_1Ctext`).
- **Разделение логики**. Класс `v8catalog` отвечает за навигацию по файлам, а логика интерпретации содержимого вынесена в отдельные функции парсинга.

## 10. Схема классов и функций

| Компонент | Назначение | Файл (примерно) |
|-----------|------------|-----------------|
| `TMainForm` | UI, точка входа, управление жизненным циклом | `MainUnit.cpp` |
| `ActionFileOpenExecute` | Обработчик события открытия | `MainUnit.cpp` |
| `v8catalog` (класс-обертка) | Чтение заголовка каталога, FAT, списка файлов | `uMain.cpp` / отдельный юнит |
| `v8file` | Представление отдельного файла внутри каталога | `uMain.cpp` |
| `get_treeFromV8file` | Извлечение структуры из файла `version` | `ParserUnit.cpp` (или аналог) |
| `parse_1Ctext` | Парсинг текстового формата 1С в объектное дерево | `ParserUnit.cpp` |
| `TFileStream` | Низкоуровневое чтение байт | VCL (`Classes.hpp`) |

## 11. Примерный псевдокод основного цикла

```cpp
// Упрощенное представление логики из v8_reader

void OpenFile(String fileName) {
    // 1. Создание объекта каталога
    auto catalog = new v8catalog(fileName);
    
    // 2. Инициализация (чтение заголовка и FAT)
    // Внутри определяется формат (15/16) и смещения
    catalog->initialize(); 
    
    // 3. Проход по таблице файлов (FAT)
    for (int i = 0; i < catalog->FileCount; i++) {
        auto file = catalog->GetFile(i);
        
        // Чтение имени из заголовка файла
        String name = file->ReadHeaderName();
        
        // Добавление в список для UI
        TreeView->AddNode(name, file);
    }
    
    // 4. Спецобработка для имени конфигурации
    auto versionFile = catalog->FindFile("version");
    if (versionFile) {
        auto tree = get_treeFromV8file(versionFile);
        String configName = ExtractConfigName(tree);
        MainForm->Caption = configName;
    }
}

// При клике на узел дерева:
void OnNodeClick(Node* node) {
    auto file = (v8file*)node->Data;
    
    // Чтение данных файла
    auto data = file->ReadData();
    
    // Конвертация в текст и парсинг
    String text = BinaryToText(data);
    auto metaTree = parse_1Ctext(text);
    
    // Отображение в редакторе
    Editor->Text = metaTree->ToString();
}
```

## 12. Заключение

Алгоритм `v8_reader` базируется на концепции **каталога с таблицей файлов (FAT)**, а не на потоковой распаковке цепочек блоков. Понимание этой архитектуры критически важно для корректного переноса функционала на C++17/Qt6. Основные усилия при портировании должны быть направлены на эмуляцию работы с FAT и реализацию парсера текстового формата 1С.
